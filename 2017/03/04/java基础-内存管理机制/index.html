java基础-内存管理机制

Java虚拟机在执行Java程序的过程中会把它管理的内存划分成若干个不同的管理的区域，这些区域有各自的用途，以及创建和销毁的时机。

程序计数器

定义：

是一块较小的内存空间，他可以看成当前线程执行字节码的行号指示器。在虚拟机的概念模型里，字节码解释器就是通过改变这个计数器的值来选取下一条需要执行的字节码指令。分支、循环、跳转、异常处理、线程恢复等基础功能都需要依赖这个计数器

存在的原因

由于Java虚拟机的多线程是通过线程轮流切换来完成的，也就是说无论何时一个处理器只会执行一个线程，为了可以回到原线程正确的位置，所以每个线程都有一个独立的程序计数器。各线程之间的程序计数器不互相影响，独立存储，我们管这片区域叫“线程私有”的区域。

存储内容

如果说当前线程执行的是一个java方法，那么此时计数器记录的是这在执行的虚拟机字节码指令地址，如果说执行的是虚拟机native方法，那么计数器的值为空（udefined）。

注：

程序计数器这片内存区域是整个虚拟机唯一一个不会出现OOM的区域。

虚拟机栈（也就是常说的栈）

定义：

也是线程私有的一块区域
描述的是java方法执行的内存模型：每个方法执行的时候都会创造一个栈帧。这个栈帧用于存储局部变量表、操作数栈、动态链接、方法出口等信息。我们谈论的栈通常是局部变量表这一块。
局部变量表存放了编译时期可知的各种基本数据类型，和对象引用（指向一个对象的引用指针）和returnAddress类型（指向了一条字节码指令的地址）
其中64位长long和double类型都会占用2个局部变量空间，其余的占用1个。局部变量表都是在编译时期就已经确定了空间的使用大小，所以当进入一个方法时，这个方法在栈中需要多大内存就已经确定了，在运行时期不会更改。
java虚拟机规范中这个区域可能出现两种异常
StackOverFlowError：线程请求的栈深度大于当前虚拟机所允许的深度（
由于当前线程的栈满了，通常是循环调用过多导致）
OutOfMemoryError：当前栈动态申请内存时没有办法申请到足够的内存

本地方法栈

与虚拟机栈发挥的作用相当类似他们之间的区别
本地方法栈：为虚拟机本身使用到的native方法服务
栈（虚拟机栈）：是为了虚拟机执行java方法服务。
它也会有StackOverflowError和OutOfMemoryError这两种异常。

java堆

对于大多数应用来说，堆的内存空间应该是最大的。这片内存区域存在的目的就是存放实例对象。是所有线程共享的一片区域，在虚拟机启动的时候创建。
此区域存在的目的就是存放实例，所有对象的实例和数组都在堆上分配。
java堆可以是物理上不连续的空间，只要逻辑上是连续的即可。
堆可以抛出OOM异常。

方法区

也是所有线程共享的一片区域
用来存储已经被虚拟机加载的类信息、常量、静态变量、即时编译器编译后的代码等数据。

可以抛出OOM异常

运行时常量池

运行时常量池是方法区的一部分
class文件除了有类的版本、字段、方法、接口等描述信息外，还有一项信息是常量池。他主要用处存放编译期生成的各种字面量和符号引用，这部分将在类加载后进入方法区的运行时常量池中存放。
运行时常量池对于class文件中的常量池的一个重要特征是具有动态性，java不要求常量一定在编时期才能产生，也就是说不是只有class文件中常量池才能进入到运行时常量池，运行期也有可能产生新的常量放到运行时常量池中。
